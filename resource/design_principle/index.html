# SOLID设计原则

# 单一职责原则 (Single Responsibility Principle)

单一职责原则指一个类应该有一个单一的职责。遵循单一职责原则的一个好的做法是在备注中列出这个类的职责，你可以用这种方法提醒自己和其他人这个类的用途并且让这个类的职责最小化。

app/EmptyGarden.php

/**
 * @purpose
 * 提供一个有土壤的能种植和产出庄稼的空的农场
 */
class EmptyGarden
{
    private $width;
    private $height;
    
    public function __construct($width, $height)
    {
        $this->width = $width;
        $this->height = $height;
    }
    
    public function items()
    {
        $numbersOfSpots = ceil($this->width * $this->height);
        return array_fill(0, $numbersOfSpots, '一戳土壤');
    }
}

你也可以把items()方法叫做harvest()，但这就意味这一个农场可以收割它自己。正常地应该由农场主或收割工人从农场里面收割农产品。农场仅仅提供庄稼生长的空间。农场自己收割自己意味着给农场添加了其他的职责。

不要使用职责单一原则太过头。一个类改变的可能性越小，这些设计原则越不重要。如果EmptyGarden类真的永远不会改变，给类加上harvest()方法也是可以的。

使用设计原则时也需要有权衡。如果维护一个大点的类比维护很多个分离的小类更加容易，那么不必执着于设计原则的形式。

# 开闭原则 (Open/Close Principle)

$garden = new App\CornGarden(10, 10);
$garden->items();

为什么要创建CornGarden？难道这比传递一个string Corn参数给Garden class会更简单吗？表面上看，它确实更容易，它会有更少的类需要追踪。但最终你每次想添加一个新的类型的garden时都需要传递一些参数并对其做一些条件逻辑处理。 OPEN/CLOSED原则指出类应该对于扩展是开放的，但是对于修改是封闭的。 如果每次需要创建一个新的类型的garden时都需要修改这个Garden类，这就违背了open/closed原则。items()方法返回不同的依赖于garden类型的items。注意以下的方法可以轻松的让你不需要修改原始的Garden类去添加很多不同的garden。 app/CornGarden.php

class CornGarden Extends EmptyGarden
{
    public function items()
    {
        return array_fill(0, $this->>width * $this->>height, '有机土壤');
    }
}

# 里氏替换原则 (Liskov Substitution Principle)

里氏替换原则注意是基于PHP是鸭式类型语言。它没有严格的变量类型。然而，PHP有类型提示。 程序里的对象可以被他们的子类型实例替换，并且不会改变程序的正确性。 当你创建EmptyGarden对象时，如果你给garden的width和height传入一个string或者class或者负数时，将会发生什么？ > new EmptyGarden("foo", -1);

这将很糟糕。它会导致程序被破坏。你是否愿意程序有这样的风险？也许你可以在这里做一点小小的重构，而不是让你的garden依赖于width和height，让它依赖于一个PlotArea。这可以是一个占据一定空间的一个接口。

src/EmptyGarden.php

public function __construct(use App\PlotArea $plot)
{
    $this->>plot = $plot
}

app/PlotArea.php

namespace App;
interface PlotArea
{
    public function totalNumberOfPlots();
}

PlotArea告诉你有多少可以用于种植的区域。一个圆形的garden可能有20块地(plots)并且半径为10英尺，一个长方形garden可能有10x10英尺的40块地。请注意它是一个接口而不是混合的class。 现在你需要改变EmptyGarden和CornGarden。 app/EmptyGarden.php

public function __construct(Plot Area $plot)
{
    $this->>plot = $plot;
}
public function items()
{
    $numberOfPlots = $this->plot->totalNumberOfPlots();
    return array_fill(0, $numberOrPlots, 'handful of dirt');
}

但是现在你怎样调用EmptyGarden呢？PlotArea不是一个class，它是一个interface。难道你要像如下创建一个RectangleArea class来implements PlotArea?

app/RectangleArea.php

use App\PlotArea;
class RectangleArea implements PlotArea
{
    private $width;
    private $height;
    public function __constuct($width, $height)
    {
        $this->width = $width;
        $this->>height = $height;
    }
    public function totalNumberOfPlots()
    {
        return ceil($this->width * $this->height / 2);
    }
}

现在你可以用tinker来测试你的class

php artisan tinker
$garden = new App\EmptyGarden(new App\RectangleArea(10, 10));
$garden->items();

另一个破坏里氏原则的是从相同的方法返回不同类型的结果。在强类型的语言如Java，这并没有什么问题。但在PHP中它可能是一个问题。 设想如果你调用$garden->items(), 你希望返回一个什么类型的数据？array？难道不是CornGarden而要返回一个string？这将会完全凌乱。它也破坏了里氏原则。 不幸的是，在php5中你不能在方法里面去设置一个返回的类型提示。好消息是返回类型提示是php7的一个新的特性。这样你可以遵循以下的语法：

public function items() : array
{ ... }

如果你还在用php5，你仍然可以使用文章来注释@return。虽然它不会帮助你做任何的编译，但它可以帮助其他查看代码的开发者。如果你正在用php7，我绝对推荐用类型提示。如果你在子类中的方法不小心返回了不同类型的结果，它会很容易破坏里氏原则并给予提示。

# 接口隔离原则 (Interface Segregation Principle)

如果非常有用的东西，它必将变动，对吗？你想添加给garden添加更多的功能，生长植物。你在农场里面播种，并且经过一系列工作之后，那些种子生长并结出果实。

app/GardenInterface.php

namespace App;
{
    public function grow($advanceNumberOfDays);
}

这样很好，然而农场也有施肥、浇水、播种、除害并且依赖于天气例如阳光和雨水。这样对于一个class就有很多的职责，让我们看看给garden添加很多方法后是什么样子。

app/GardenInterface.php

namespace App;
{
    public function grown($advanceNumberOfDays);
    public function weed($pickOutPercentage);
    public function pestAttack($attachFactor);
    public function water($inGallons);
    public function sunshine($radiationalLevel);
    public function fertilize($type, $amount);
    ...
}

请注意这个GardenInterface变得越来越大，因此任何具体class的职责将implements GardenInterface。这意味着单一职责原则也可能被违背。当你违背SOLID原则其中一个时，其他的原则也有可能被违背。 SOLID原则是互相融合协同工作的。主要的问题是你正在将大陆的功能添加到一个class中，给管理这个class带来了更大的困难。 我们是如此容易沉迷于模型和对于实体的思考，以至于我们忘记面向对象的程序不仅仅是对“事物”创建对象。封装行为也是OO(object-oriented)设计的一个强大的部分。 如果你希望只有一个grow()方法怎么办？你是否将implement所有其他的方法仅仅为了得到grow()这个方法？你可以生成一个garden像很多子接口的集合。

src/GardenInterface.php

interface GardenInterface implements GrowableInterface, WeedableInterface, ...
{
}

由很多小的接口组成的主接口的确具有更多的灵活性，因为它将允许你从那些小的接口里面选择。但是它并没有解决你的garden变得越来越复杂的问题。最终，你结束了添加更多的类来解决这个问题的方法。现在，你仅仅给EmptyGarden class创建空的方法就行了。 保持你的接口小巧以遵循接口分离原则：很多特定接口比一个通用的接口更好。

# 依赖反转原则 (Dependency Inversion Principle)

依赖反转原则规定高层级的类依赖于抽象而不是具体。这是什么意思呢？为了回答这个问题，回想一下当你给你的garden定义PlotArea的时候，是否你做了下面的事情：

app/Garden.php

public function __construct(RectangleArea $plot)
{
    $this->plot = $plot;
}

这样做就会强迫你用矩形来创建每一个garden。如果你需要其他形状的garden，这样做根本不起作用。为了避免不够灵活的问题，你可以用一个抽象接口(PlotArea interface)来代替一个具体的类(RectangleArea class). 这种形式的依赖反转作为依赖注入被熟知，并且被注入到一个实施了特定接口的class。 另一个处理依赖反转冲突的方法是你开始看代码中的关键词的时候，设想你是否已经在items()方法内创建了新的长方形区域。

app/Garden.php

public function items()
{
    $numberOfPlots = new RectangleArea;
    return array_fill(0, $numberOfPlots, 'handful of dirt');
}

在这个例子中，EmptyGarden class是高层级的类并且依赖与低层级的类RectangleArea。

有时候在你的应用中，你可能将用new这个词来创建一个对象，这很好，然而它意味着这个类被捆绑到其他的类并且产生了强依赖关系。用new创建对象并没有什么错误，但是在一个错误的地方用了它，将导致程序很脆弱，被捆绑的程序做测试和维护都会更具挑战性。我将继续陈述高层级代码和工厂(你可以在后面的factories章节了解更多)。 在软件工程中，有一个说法：低耦合，高内聚。耦合是降低一个类对另一个类的依赖程度。内聚是一个类中的元素在多大程度上属于另一个类。

假如你的类是一个岛屿，你希望内部的一切都有序工作。好的经济，低的犯罪等等。如果你的岛屿依赖与其他国家，你希望保持依赖程度的最小化。为什么？设想如果一个你依赖的国家在facebook上对你不友好，导致你的经济破产。现在你的人民正在忍饥挨饿。这并不cool。完全不cool。因此依赖较少的外部国家是一个低内聚高耦合的例子。最好的岛屿是岛屿人员的的需求可以由他们自行满足并且不依赖与他们之外的国家。 为了减轻耦合和内聚问题，遵循高层级的类不应该依赖与低层级的类的惯例。而是依赖于抽象例如：接口、抽象类等。

你是怎么知道高层级类依赖于低层级类。

描绘一下一个高层级的类是一个正在指挥音乐会的音乐家。低层级的类是拿着乐器参与演奏音乐会的人。指挥者在精心策划并指挥低层级的类，产生出美妙的音乐。 然而，设想音乐家依赖于一个具体的表演者：Bob大号演奏者。如果Bob因流行感冒生病，音乐师必须关闭这个大型的音乐会。如果他不依赖与Bob，而是依赖于抽象：大号演奏者。队伍中的Fred，一个不同类型的大号演奏者，就这样今天晚上的表演被拯救了！音乐师并不担心是谁Bob，Fred，或者Sally来演奏大号，因为他们都是合格的大号演奏者。这就是在最合适的地方的依赖注入。 有一点需要注意：依赖反转和依赖注入不是一回事。另外一个方法实现依赖反转是使用控制容器反转。

app/Maestro.php

class Maestro
{
    public function conduct($song)
    {
        $tubaPlayer = app()->make('tuba.player');
        $clarinetPlayer = app()->make('clarinet.player');
        foreach([$tubaPlayer, $clarinetPlayer] as $player)
        {
            $player->play($song);
        }
    }
}

注意你在这里不依赖于任何的类。相反你让app()->make()提供了演奏需要的tuba player和clarinet player。他们可以很容易被包裹在服务容器。

app/Providers/PlayerServiceProvider.php

如果Composer是Laravel的骨头，那么服务容器就是大脑。你将在后面的章节学到更多关于Laravel的服务容器。

# 总结

  1. 单一职责原则

    不要把所有的鸡蛋放到一个篮子里。一个类应该只有一个原因去让他改变。

  2. 开/闭原则

    不要重复的去更改一个类。如果你发现正在发生这种事情，抽象出正在发生改变的东西。

  3. 里氏替换原则

    在子类中重新的方法中返回与父类方法相同的类型，相同的类型也适用于方法的参数。保持一致。

  4. 接口隔离原则

    不要给接口创建过多的方法。这是在一个地方做太多事的朕兆。

  5. 依赖反转原则

    依赖与接口和抽象类好过具体实现类。这将更具有灵活性。

  6. 封装变化的内容

    将变化的内容抽离于你的应用。例如，如果一个Mailer class将永远不会变化，不要急于写很多的抽象：关注将会变化的东西。
